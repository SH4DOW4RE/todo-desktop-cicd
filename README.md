# Todo Desktop CICD

Todo Desktop CICD is an Electron desktop client for managing todos through the Shadoweb Todo API. It provides a custom desktop interface for folders, notes, tags, parent relationships, authentication, and account maintenance, with automated tests, coverage gates, native release builds, and GitHub-based update checks.

## Features

- Create, edit, sort, change the status of, and delete todos.
- Organize todos by nested folders.
- Add tags with autocomplete that completes the tag currently being typed.
- Link todos to multiple existing parent notes while preventing circular dependencies.
- Authenticate against the remote Todo API and keep the JWT in Electron's encrypted storage.
- Clear account-owned todos and folders from the account modal with confirmation.
- Display the application version and whether the app is packaged or running in development.
- Show update status in the title bar on packaged builds.
- Install updates automatically on Windows and direct Linux/macOS users to the latest GitHub Release.

## Technology Stack

- Electron 42
- Electron Forge 7
- Node.js 24 for CI tests and coverage
- Node.js 22 for release builds
- CommonJS JavaScript
- Native `node:test` test runner
- Electron Forge makers:
  - Windows: Squirrel installer
  - Linux: Debian and RPM packages
  - macOS: ZIP application archive

## Requirements

- Node.js 22 recommended, especially for packaging. CI tests and coverage run on Node.js 24.
- npm.
- A reachable Todo API:
  - Production: `https://todo-api.shadoweb.fr`
  - Debug mode: `http://127.0.0.1:4040`

## Setup

```bash
npm ci
```

For local API development, start the API on `127.0.0.1:4040` and launch the app with:

```bash
SHADOWEB_DEBUG=1 npm start
```

On Windows PowerShell:

```powershell
$env:SHADOWEB_DEBUG = "1"
npm start
```

## CI

The CI workflow runs on `ubuntu-latest` with Node.js 24 and performs:

1. Dependency installation with `npm ci`.
2. Unit tests with `npm test`.
3. Coverage enforcement with `npm run coverage`.

Coverage is enforced at 90% for lines, branches, and functions over the API modules.

## Release Process

The release workflow is triggered by tags matching `v*`, for example:

```bash
git tag -a v1.0.1 -m "Release v1.0.1"
git push origin v1.0.1
```

The workflow:

1. Runs tests and coverage on `ubuntu-latest` with Node.js 22.
2. Builds installers on their native operating systems:
   - Windows x64 on `windows-latest`
   - Linux x64 on `ubuntu-latest`
   - macOS arm64 on `macos-latest`
3. Uploads the generated Windows Squirrel files, Debian/RPM packages, and macOS ZIP archive.
4. Publishes the artifacts to a GitHub Release.

Manual release dispatch is also available from GitHub Actions and requires an explicit tag input.

The version in `package.json` and the Git tag should match. Windows updates require the release to include `Setup.exe`, `RELEASES`, and the full `.nupkg` generated by Squirrel.

### macOS first launch

The macOS build is currently ad-hoc signed but not notarized.

On first launch, macOS may block the app. To open it:

1. Try to open **Todo Desktop CICD**.
2. In the warning dialog offering **Move to Trash** or **Done**, click **Done**. This is required for the app to appear in the blocked-app settings.
3. Open **System Settings**.
4. Go to **Privacy & Security**.
5. Scroll to the security section containing the blocked **Todo Desktop CICD** message.
6. Click **Open Anyway**.
7. Confirm that you want to open the app.

After that, the app should launch normally.

## Auto Updates

Update checks only run in packaged builds. Development sessions started with `npm start` or `npm run dev` do not check for updates.

### Windows

Windows uses Electron's built-in `autoUpdater` with the `update.electronjs.org` feed:

1. The app checks for a newer GitHub Release at startup and once per hour.
2. Squirrel downloads the update in the background.
3. The title bar displays the current update state.
4. When the download finishes, the user can restart immediately or install it later.

The installer and update files remain hosted in GitHub Releases. `update.electronjs.org` acts as the feed adapter between Electron and those release assets.

### Linux and macOS

Linux and macOS use a manual update flow because these builds are not distributed with a suitable signed automatic-update installation path:

1. The app queries the latest GitHub Release at startup and once per hour.
2. If the installed version is current, the title bar briefly displays **App is up to date**.
3. If a newer version exists, the app asks whether the user wants to download it.
4. Choosing **Download** opens that release on GitHub, where the user installs it manually.

The app does not download or install Linux/macOS updates automatically.

## Security Notes

- `.env` is ignored and must not be committed.
- API tokens are stored in Electron's `safeStorage` when encryption is available.
- The renderer only accesses backend capabilities through the explicit preload bridge.
- Release publishing uses the GitHub-provided `GITHUB_TOKEN` with `contents: write`.
